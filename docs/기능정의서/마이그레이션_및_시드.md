## 1. 마이그레이션 도구 선택

| 항목 | 결정 |
|------|------|
| **도구** | **Flyway** (Community) |
| **DB** | **PostgreSQL 15+** (`DB스키마_초안.md` 권장) |
| **형식** | 순차 SQL 파일 `db/migrations/V{nnn}__description.sql` |
| **대안** | Liquibase (XML/YAML) — 팀에 DB 변경 리뷰 자동화가 필요할 때; v0.1은 SQL 단순성 우선 |

### Flyway 선택 이유

1. **SQL-first** — 스키마 초안이 이미 PostgreSQL DDL로 표현 가능.
2. **버전 추적** — `flyway_schema_history`로 dev/prod 적용 이력 분리 (`배포_아키텍처.md` §4 dev/prod 2환경).
3. **CI 연동** — `flyway migrate` 한 줄로 파이프라인에 삽입 용이 (백엔드 착수 후).
4. **Alembic 제외** — 런타임은 **Node + Drizzle**(`백엔드_스택_결정.md`); DDL은 Flyway만 유지.

### 버전 관리 전략

| 규칙 | 내용 |
|------|------|
| 파일명 | `V001__initial_schema.sql`, `V002__add_foo.sql` — **한 번 적용된 파일은 수정 금지** |
| 롤백 | Flyway Community는 down migration 미지원 → **수정은 새 `V00n` forward migration** |
| 환경 | dev·prod **동일 migration 경로**, DB 인스턴스만 분리 (체크 233) |
| 시드 | Flyway **외부** — `db/seed/dev_seed.sql` 수동/스크립트 (prod 시드 없음) |
